# AGENTS.md

## Project overview

このプロジェクトは、OODA型ケース見立て支援ツールを実装するためのWebアプリケーションである。

このアプリは、公式記録、診断支援、請求業務、個別支援計画の法定様式を代替するものではない。目的は、現場で観察された事実をもとに、支援仮説を立て、小さな支援を一つ試し、その反応を記録して見立てを更新することである。

中核設計は OODA である。

- Observe: 見えた事実を記録する。
- Orient: 事実から複数の支援仮説を立てる。
- Decide: 次に試す低リスクの小さな支援を一つ決める。
- Act: 実施内容と反応を記録し、仮説を更新する。

## Product principles

必ず以下の思想を守ること。

1. 公式記録を書く道具にしない。
   - 個別支援計画、請求、国保連、法定記録の本体は作らない。
   - 既存の業務管理システムへ転記しやすい一段落要約は作ってよい。

2. 診断名や人格評価から入らない。
   - 「発達障害だから」「性格が弱い」「やる気がない」などの判断を出さない。
   - 場面、直前の出来事、行動、直後の結果、環境、対人文脈、身体状態、本人の選好から支援仮説を作る。

3. Observe と Orient を明確に分ける。
   - Observe には事実を書く。
   - Orient には推測、仮説、未確認点を書く。
   - UI上でも、事実欄と仮説欄を混ぜない。

4. Orient を厚くする。
   - 仮説は最低2つ、最大3つ出せるようにする。
   - 反証、未確認点、追加で見るべき点を入力できるようにする。
   - 単一原因に決めつける設計にしない。

5. Decide と Act は小さく速く回す。
   - 一度に試す支援は原則1つにする。
   - 複数介入をまとめて実施すると、何が効いたか分からなくなるため避ける。
   - 指標を事前に決めてから試す。

6. AI補助は「答え」ではなく「考え漏れを減らす補助」として扱う。
   - 許可するAI補助:
     - 不足している観察項目の指摘
     - 仮説の別案提示
     - trait語、人格評価語の警告
     - 支援を一つに絞る提案
     - 類似場面の提示
   - 禁止するAI補助:
     - 診断名の推定
     - リスクの自動断定
     - 効果保証つきの支援提案
     - 本人の意思や価値観の代弁
     - 医療判断

7. 個人情報と安全性に配慮する。
   - 利用者名は表示名またはケースIDで扱えるようにする。
   - 生成AIへ送るデータには、氏名、住所、電話番号、具体的な勤務先などの不要な個人識別情報を含めない。
   - 緊急リスクがある場合は、アプリ内の通常フローではなく、事業所の緊急対応フローに接続する表示を出す。

## Required domain categories

見立てカテゴリーは、診断カテゴリではなく、支援で変えうるメカニズム別に持つこと。

最低限、以下の10カテゴリーを実装する。

1. 起動・開始の困難
2. 見通し・変化への弱さ
3. 理解・段取り負荷
4. 感覚・環境負荷
5. 対人・評価負荷
6. 失敗回避・不安
7. 援助要請の障壁
8. 強化・回避の機能
9. 身体状態
10. 本人の選好・価値

## Required observation fields

Observe画面には、少なくとも以下の入力群を持たせる。

- 場面の基本情報
  - 日時
  - 場所
  - プログラム名
  - 開始前、実施中、終了前などのタイミング
  - 同席者

- 直前の出来事
  - 指示
  - 予定変更
  - 待機
  - 休憩終了
  - 修正指摘
  - 対人やり取り
  - 移動直後

- 行動の事実記述
  - 開始できない
  - 手が止まる
  - 席を離れる
  - 返事のみ
  - 質問しない
  - 再確認を繰り返す
  - 顔を伏せる
  - 自由記述

- 行動の測定
  - 開始までの時間
  - 止まった時間
  - 声かけ回数
  - 再開までの時間

- 直後の結果
  - 課題が外れる
  - 休憩になる
  - 安心を得る
  - 注目を得る
  - 修正完了
  - 対人離脱

- 作業条件
  - 新規または既知
  - 工程数
  - 多重課題
  - 時間制限
  - 完成基準の明確さ
  - 難易度

- 環境・感覚
  - 音
  - 動線
  - 視覚刺激
  - 照明
  - におい
  - 温度
  - 混雑
  - 個人スペース

- 対人文脈
  - 相手
  - 関係性
  - 1対1または複数
  - 評価場面か
  - 口頭、書面、実演指示
  - フィードバックの調子

- 身体・心理の手掛かり
  - 眠気
  - 疲労
  - 頭痛
  - 痛み
  - 空腹
  - 服薬後
  - 焦り
  - 反芻
  - 涙
  - 過緊張

- 保護因子・好み
  - 有効だった支援
  - 好む声かけ
  - 静かな場所
  - 見本
  - 選択肢
  - 本人の価値や目標

- リスク・緊急度
  - 自傷他害
  - 急な逸脱
  - 解離様
  - 強いパニック
  - 医療確認要否
  - 緊急度低、中、高

各項目には「不明」「未確認」を選べるようにする。無理に埋めさせない。

## Required screens

最低限、以下の画面を実装する。

1. ホーム
   - 今日の未レビュー観察
   - 試行中の仮説
   - 期限が来た見直し
   - 最近の成功条件

2. Observe入力画面
   - 60〜90秒で入力できることを重視する。
   - 最初に自由記述欄を置く。
   - その下にタグ、数値、チェック項目を置く。
   - 事実ではない表現が入った場合は警告する。

3. Orient画面
   - 観察カードから仮説カードを作る。
   - 仮説候補を1〜3個入力できる。
   - 推奨として最低2個を促す。
   - 各仮説には、根拠、反証、未確認点、追加で見る点を入力する。
   - 診断名や人格評価語に寄せない。

4. Decide画面
   - 次に試す支援を一つ選ぶ。
   - 狙う変化を入力する。
   - 測る指標を選ぶ。
   - 観察期限を設定する。
   - 複数支援を同時に選んだ場合は警告する。

5. Act画面
   - 実施内容
   - 予定通り、一部変更、未実施
   - 直後反応
   - 後続反応
   - 指標変化
   - 仮説が強まった、弱まった、保留の選択

6. 類似場面検索画面
   - 「休憩後」「予定変更後」「音が高い」「質問できない」などで過去ケースを検索できる。
   - 類似しているだけで同一原因と扱わない注意文を表示する。

7. チームレビュー画面
   - 事実列と仮説列を分けて表示する。
   - コメント欄だけの雑談画面にしない。
   - 仮説の更新履歴を残す。

8. エクスポート要約画面
   - 公式記録へ転記しやすい一段落要約を作る。
   - ただし、このアプリの出力は法定記録の代替ではないと表示する。

## Required templates

以下の3種類の支援仮説テンプレートを実装する。

### 基本型

「〔場面〕で、〔観察された事実〕が起きている。直前には〔先行条件〕があり、直後に〔結果〕が生じた。背景には〔仮説〕が影響している可能性がある。次回は〔小さな支援〕を試し、〔指標〕の変化を見る。」

### FBA型

「〔setting event〕と〔antecedent〕があると、〔behavior〕が起き、〔consequence〕につながっている。この行動は〔get/obtain〕または〔escape/avoid〕の機能を持つ可能性がある。次回は〔antecedent adjustment / replacement skill〕を試し、〔latency / duration / frequency〕を見る。」

### ACT補助型

「〔不安・思考・感覚〕を避けようとすると、〔止まる・離れる・質問しない〕が起きやすい可能性がある。本人にとって大事な〔価値・目標〕に沿って、〔ごく小さい行動〕を支え、〔再開・継続〕の変化を見る。」

## Non-goals

以下は実装しない。

- 診断名の自動推定
- 医療判断
- 障害支援区分の判定
- 請求データ作成
- 国保連連携
- 法定個別支援計画の完全自動作成
- リスクの自動断定
- 効果保証つきの支援提案
- 利用者を人格特性で分類するスコアリング

## Writing rules for generated text

生成される文章は、以下を守る。

- 事実と推測を分ける。
- 「やる気がない」「甘えている」「協調性がない」「打たれ弱い」「問題行動」などの評価語を避ける。
- 「返事はあったが、3分間キーボードに触れなかった」のように、実際に見える行動で書く。
- 「可能性がある」「未確認」「次回確認する」のように、仮説の暫定性を明示する。
- 断定しない。
- 読み上げても自然な日本語にする。
